# AGPL-3.0: юридический разбор для Олимп / NOTA LAB

> **Дисклеймер:** Я не юрист и это не юридическая консультация. Документ
> описывает технические сценарии использования AGPL-3.0 и общеизвестные
> практики комьюнити. Для финального решения — консультируйся с юристом,
> который разбирается в open-source лицензиях.

## TL;DR

✅ **Что МОЖНО без обязательств:**
- Использовать OCE для собственных нужд (внутри ООО «Олимп») — **без ограничений**
- Модифицировать код «под себя» — **можно, без обязательств**
- Деплоить на свой VPS, использовать сметчики, прорабы, бухгалтерия — **можно**
- Интегрировать через REST API с твоим ERPNext — **можно**
- Хранить там данные клиентов (НТМК, СВЕЗА) — **можно**

⚠️ **Что обязывает раскрыть код:**
- Предоставить доступ к OCE **внешним пользователям** (за деньги или бесплатно)
  — например, открыть «ОЛИМП.СМЕТА» как сервис для других подрядчиков
- Любое использование «как SaaS» внешним лицам

❌ **Что точно нельзя:**
- Продавать OCE как закрытый продукт без публикации источников
- Встраивать AGPL-код в свои продукты с проприетарной лицензией

## Что такое AGPL-3.0 простыми словами

AGPL — это GPL + закрытие «облачной дыры».

**GPL:** если ты **распространяешь** программу, ты обязан дать код.
**AGPL:** если ты **предоставляешь к программе сетевой доступ**, ты обязан дать код.

То есть, если ты поднимешь OCE на VPS и дашь логиниться **сторонним пользователям**
(подрядчикам, заказчикам, кому угодно за пределами твоего ООО), эти пользователи
имеют право потребовать у тебя исходный код всей системы — включая твои модификации.

## Сценарий 1: Внутреннее использование в ООО «Олимп»

**Кто пользуется:** Дмитрий (директор), сметчик, ПТО, гл. инженер, прорабы.
Все — сотрудники ООО «Олимп», работают по трудовым договорам.

**Юридический статус:** ✅ **Полностью чисто**.

В терминах AGPL твои сотрудники — это **внутренние пользователи юрлица**.
Не «третьи лица». Юрлицо использует софт для своих нужд.

Можно:
- Кастомизировать код под Олимп (логотип, цвета, ГЭСН/ФЕР интеграцию)
- Хранить там сметы своих клиентов (НТМК, СВЕЗА — это **твои данные**, не их)
- Не публиковать модификации
- Не показывать клиентам, что у тебя AGPL-софт под капотом

## Сценарий 2: Доступ давать прорабам субподрядчиков

**Серый сценарий.** Если ты даёшь логин в OCE прорабу субподрядчика
(физлицу или другому юрлицу), теоретически это **«предоставление сетевого доступа третьему лицу»**.

**Что делать:**
- Если прораб — оформлен как ИП/ГПХ напрямую на Олимп → ✅ внутренний пользователь
- Если прораб — сотрудник стороннего ООО → ⚠️ серая зона, лучше не давать

**Решение:** не давать прорабам субов прямой доступ в OCE.
Они работают в **ERPNext** (через Supplier Portal — это другая лицензия, GPLv3) или
получают экспортированные PDF-сметы.

## Сценарий 3: Заказчик НТМК просит дать доступ посмотреть смету

**Подход:** экспорт в PDF/Excel, отправляешь файлом по почте/Диадоку.
**Не давай НТМК логин в OCE напрямую** — это создаёт AGPL-обязательства.

## Сценарий 4: «Олимп.Смета» как сервис для других строительных компаний

**Идея:** ты сделал классную кастомизацию под Россию (ГЭСН/ФЕР, КС-2/КС-3,
российская терминология), и думаешь продать другим подрядчикам.

**AGPL-обязательства срабатывают** в полном объёме:
- Любой клиент имеет право потребовать у тебя ВЕСЬ исходный код, включая твои
  кастомизации, под той же лицензией AGPL-3.0
- Клиент имеет право взять твой код и сам перепродать его, конкурируя с тобой
- Ты не можешь продавать «proprietary edition» — это нарушение AGPL

**Варианты:**

**(А) Принять AGPL и идти в open core модель.**
- Базовый код — AGPL-3.0, бесплатно, на GitHub
- Зарабатываешь на: внедрении, поддержке, обучении, кастомизации, хостинге
- Это рабочая модель (так живут ERPNext, Mattermost, Element).
- Минус: твои конкуренты получают твой код бесплатно.

**(Б) Купить коммерческую лицензию у Артёма Бойко.**
- В README прямо написано: `For commercial licensing without AGPL obligations,
  contact info@datadrivenconstruction.io`
- Стоимость не публичная — нужно писать и согласовывать.
- Плюс: можешь делать закрытый продукт.
- Минус: разовая или ежегодная плата, риск зависимости от автора.

**(В) Не использовать OCE вообще для SaaS-направления.**
- Берёшь только идеи (модель данных, UX-паттерны)
- Реализуешь как DocType в твоём ERPNext (GPLv3, тоже copyleft, но не закрывает
  «облачную дыру»)
- Минус: 4–6 месяцев работы вместо готового решения.

## Сценарий 5: NOTA LAB и парфюмерия

OCE никаким образом не пересекается с NOTA LAB. Парфюмерный e-commerce
живёт на WordPress (GPLv2) — там своя лицензионная история, AGPL не касается.

## Рекомендованный путь для тебя

Учитывая, что:
- Сейчас задача — оптимизировать **собственные сметы** (НТМК, СВЕЗА, Прософт-Системы)
- SaaS для подрядчиков — пока не приоритет, бизнес-модель не подтверждена
- Юридическая чистота важна, но не блокирует внутреннее использование

**План:**

1. **Сейчас (0–6 месяцев):** Используешь OCE для внутренних нужд Олимпа.
   Все обязательства AGPL **не активируются**, т.к. нет внешних пользователей.
   Кастомизируешь под себя свободно.

2. **Если задумаешь SaaS (6+ месяцев):** Принимаешь решение тогда —
   либо платная лицензия у автора, либо открываешь свой код, либо
   переписываешь модули как DocType ERPNext. На этот момент у тебя уже будет
   понимание спроса.

3. **Защитная мера:** Сразу веди свои кастомизации **в отдельном модуле/папке**
   (например `oce-custom-olimp/`), который чисто отделён от основного OCE-кода.
   Это упростит будущий выход на коммерческое лицензирование, если понадобится.

4. **Технический буфер:** Изолируешь AGPL-код в отдельном контейнере с REST API.
   Твой основной ERPNext-стек не «заражается» AGPL, потому что не линкуется с OCE
   на уровне кода — только HTTP-вызовы через n8n. Это **стандартный паттерн
   изоляции AGPL-кода**, используемый в индустрии.

## Контрольный чек-лист "AGPL safe"

- [ ] OCE деплоится только во внутренней сети `olimp_network`
- [ ] Доступ к OCE — только сотрудникам Олимпа через VPN или интранет
- [ ] Заказчики и субы получают **экспортированные файлы** (PDF, Excel), не логины
- [ ] Кастомизации лежат в отдельной папке `oce-custom-olimp/`
- [ ] В кастомной папке есть копия LICENSE (AGPL-3.0)
- [ ] Сделан git fork upstream OCE, обновления подтягиваются через rebase
- [ ] Связь OCE ↔ ERPNext только через REST API (не через прямые импорты кода)
- [ ] Никакой части OCE-кода нет в коде ERPNext (изоляция)
- [ ] Если планируется SaaS — отдельная консультация с юристом

Соблюдение этого чек-листа держит тебя в **полностью легальной зоне**
для внутреннего использования и оставляет двери открытыми для любых
бизнес-моделей в будущем.
